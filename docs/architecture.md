# Архитектура системы памяти

## Обзор

AI-агент без памяти - как человек с амнезией. Каждая сессия начинается с чистого листа. Система памяти OpenClaw решает эту проблему через 5 слоёв:

```
┌─────────────────────────────────────────────┐
│            Working Memory                    │
│   (системный промпт: MEMORY.md, AGENTS.md)  │
│   Загружается в КАЖДУЮ сессию               │
├─────────────────────────────────────────────┤
│            Episodic Memory                   │
│   (daily notes, session history, handoff)    │
│   Конкретные события и контексты            │
├─────────────────────────────────────────────┤
│            Semantic Memory                   │
│   (memory/core/, vectorDB)                   │
│   Факты и знания, доступ через memory_search │
├─────────────────────────────────────────────┤
│            Procedural Memory                 │
│   (skills/, memory/decisions/)               │
│   Как делать, правила, решения              │
├─────────────────────────────────────────────┤
│            Handoff Layer                     │
│   (handoff.md, BOOTSTRAP.md)                 │
│   Передача контекста между сессиями         │
└─────────────────────────────────────────────┘
```

## 1. Working Memory (рабочая память)

**Файлы:** MEMORY.md, AGENTS.md, IDENTITY.md, USER.md, SOUL.md, TOOLS.md

Загружаются в системный промпт каждой сессии. Агент "видит" их всегда. Поэтому MEMORY.md должен быть коротким (< 3000 символов) - каждый символ стоит токенов.

**MEMORY.md** - сводка ключевых фактов. Кто я, кто владелец, текущие приоритеты, ключевые решения. Обновляется еженедельным консолидатором.

## 2. Episodic Memory (эпизодическая память)

**Файлы:** memory/YYYY-MM-DD.md (daily notes), session history

Daily notes пишутся кроном 2 раза в день. Содержат: что было, решения, инсайты. Через 14 дней архивируются, через 90 - удаляются.

Session history хранится в OpenClaw автоматически. Доступ через `sessions_history()`.

## 3. Semantic Memory (семантическая память)

**Файлы:** memory/core/ (вечные факты), vectorDB (main.sqlite)

Core - это факты которые не устаревают: профили, документы, постоянные правила. Векторная база индексирует ВСЕ файлы памяти и предоставляет hybrid search (70% vector + 30% BM25).

Агент ищет через `memory_search("запрос")` - получает релевантные фрагменты.

## 4. Procedural Memory (процедурная память)

**Файлы:** skills/ (скиллы), memory/decisions/ (решения)

Скиллы - это "как делать" (инструкции для конкретных задач). Decisions - это "что решили" (правила и выводы из опыта). Оба типа не устаревают.

## 5. Handoff Layer (передача контекста)

**Файлы:** memory/handoff.md, BOOTSTRAP.md

Самый критичный слой. Когда контекст компактифицируется (сжимается) - старые сообщения исчезают. Handoff сохраняет всё что нужно для продолжения работы. BOOTSTRAP.md - инструкция "что делать после пробуждения".

## Компактификация (safeguard mode)

Когда контекст агента приближается к лимиту окна, OpenClaw запускает компактификацию:

1. **memoryFlush** срабатывает при softThresholdTokens (8000 токенов до лимита)
2. Агент записывает контекст в handoff.md (формат: Тема, Решения, TODO, Файлы, Контекст, Черновики)
3. OpenClaw сжимает историю, оставляя reserveTokensFloor (25000 токенов)
4. При следующем сообщении - агент выполняет BOOTSTRAP.md, читает handoff и продолжает

**safeguard mode** - самый безопасный. identifierPolicy: "strict" - не выбрасывает из контекста важные идентификаторы (имена, пути, ключи).

## Потоки данных

```
Владелец пишет
     │
     ▼
[Сессия агента]
     │
     ├── memory_search() → vectorDB → релевантные фрагменты
     ├── read handoff.md → текущий контекст
     ├── read daily note → контекст дня
     │
     ▼
[Ответ агента]
     │
     ├── Крон 3x/день → handoff.md (автосохранение)
     ├── Крон 2x/день → daily note (дневник)
     ├── Крон 1x/нед → consolidator (паттерны → core/, decisions/)
     └── Крон 1x/ночь → cleanup (архивация, ротация)
```
