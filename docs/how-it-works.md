# Как работает каждый компонент

## BOOTSTRAP.md

**Когда грузится:** при старте каждой сессии (входит в workspace files OpenClaw).

**Что делает:**
1. Агент читает handoff.md - восстанавливает контекст
2. Читает daily note - контекст дня
3. Сверяет с sessions_history - handoff актуален?
4. Если handoff устарел - ориентируется на sessions_history
5. Продолжает работу или молчит (NO_REPLY)

**Зачем:** после компактификации агент "просыпается" без памяти о предыдущем разговоре. BOOTSTRAP.md - его инструкция как вспомнить.

## handoff.md

**Формат (фиксированный):**
- Тема - над чем работали
- Решения - что решили и почему
- TODO - незавершённые задачи
- Файлы - с чем работали (точные пути)
- Контекст - факты для продолжения
- Черновики - незаконченный текст/код ЦЕЛИКОМ

**Обновляется:**
- Автоматически 3x/день (крон auto-handoff)
- Перед компактификацией (memoryFlush)
- Вручную (агент пишет при завершении задачи)

**Правила:**
- Не переносить предположения - только факты
- Точные пути, не "тот файл"
- Суть решений, не номера сообщений
- Черновики ПОЛНОСТЬЮ (после компактификации старые сообщения пропадут)

## memoryFlush

**Когда срабатывает:** когда контекст достигает softThresholdTokens (8000 токенов до лимита).

**Что делает:** OpenClaw отправляет агенту промпт "запиши контекст в handoff.md" ПЕРЕД тем как начнёт сжимать историю.

**Параметры:**
- `softThresholdTokens: 8000` - за сколько токенов до компактификации срабатывать
- `prompt` - текст промпта (формат handoff)

**Важно:** memoryFlush срабатывает ДО компактификации. Если агент не успел записать - контекст потеряется.

## Векторный поиск (memory_search)

**Как работает:**
1. Все файлы памяти индексируются (text-embedding-3-small)
2. При `memory_search("запрос")` - hybrid search: 70% vector similarity + 30% BM25 (текстовый поиск)
3. Возвращает TOP-N релевантных фрагментов с путями и строками

**Настройки:**
- `provider: "openai"` - провайдер эмбеддингов
- `model: "text-embedding-3-small"` - модель (дешёвая и быстрая)
- `vectorWeight: 0.7` - вес семантического поиска
- `textWeight: 0.3` - вес текстового поиска
- `sessionMemory: true` - индексировать историю сессий

**Когда использовать:** когда агенту нужно вспомнить что-то из прошлых разговоров, решений, проектов. Агент вызывает memory_search автоматически перед ответами о прошлом.

## Кроны

### auto-handoff (3x/день)
- 12:00, 18:00, 23:00
- Записывает текущий контекст в handoff.md
- Модель: Sonnet (экономия токенов)
- Если ничего не происходило - NO_REPLY

### auto-diary (2x/день)
- 13:00, 22:00
- Пишет/дополняет daily note (memory/YYYY-MM-DD.md)
- Формат: Что было, Решения, Инсайты
- Максимум 200 слов на обновление

### consolidator (1x/неделю)
- Воскресенье 03:00
- Читает daily notes за неделю
- Извлекает паттерны → memory/decisions/, memory/core/
- Проверяет MEMORY.md на размер (< 3000 символов)
- Дополняет, не перезаписывает

### night-cleanup (1x/ночь)
- 03:30, bash скрипт (0 токенов)
- Архивирует daily notes > 14 дней
- Удаляет архивы > 90 дней
- Ротирует логи > 10 МБ
- Чистит старые сессии > 30 дней
- Очищает embedding_cache в SQLite

## Жизненный цикл daily note

```
День 0:   Создаётся кроном auto-diary
День 1-13: Дополняется кроном (утро/вечер)
День 7:   Консолидатор извлекает паттерны → core/, decisions/
День 14:  night-cleanup архивирует в archive/daily/
День 90:  night-cleanup удаляет из архива
```
